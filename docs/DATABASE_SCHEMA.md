## 1. Overview

### Purpose

The ScouterFRC database stores every artifact produced by the automated FRC scouting pipeline: event metadata fetched from The Blue Alliance (TBA), robot performance records generated by the computer vision pipeline, manually entered scout observations, user accounts, and the audit trail of all external syncs.

### Design Principles

| Principle | Implementation |
|---|---|
| **Surrogate keys** | Every table uses a `SERIAL`/`BIGSERIAL` primary key; natural keys (TBA event key, team number) are enforced as `UNIQUE NOT NULL` rather than PKs |
| **Immutable natural keys** | TBA keys (`tba_event_key`, `tba_match_key`) are never mutated once inserted |
| **JSON for schema-flexible data** | `JSONB` columns store the 3×3 perspective-transform matrix, scout observation action payloads, and TBA sync diff snapshots |
| **Generated columns** | `total_score_contribution` and `success_rate` are `GENERATED ALWAYS AS … STORED` to avoid application-layer recomputation |
| **Explicit phase tracking** | `CHECK` constraints on `phase` columns restrict values to `('auto','teleop','endgame')` across all phase-aware tables |
| **Timezone-aware timestamps** | All timestamps use `TIMESTAMPTZ` (UTC) |

### Phase-Based Evolution

| Phase | New Models | Focus |
|---|---|---|
| **Phase 1** | EVENT, TEAM, MATCH, ALLIANCE, ROBOT_PERFORMANCE, USER, SCOUTING_OBSERVATION, SYNC_LOG | Core MVP: data collection, API, dashboard |
| **Phase 2** | SCORING_EVENT, MOVEMENT_TRACK, PHASE_STAT | Computer vision pipeline output and analytics |
| **Phase 3** | ORGANIZATION, TEAM_ORGANIZATION | Multi-tenancy and community features |

## 2. Core Models (Phase 1)

### 2.1 EVENT

Represents a single FRC regional, district, or championship event.

**Table:** `event`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `event_id` | `SERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `tba_event_key` | `VARCHAR(20)` | `UNIQUE NOT NULL` | — | The Blue Alliance event key (e.g., `2025calb`) |
| `name` | `VARCHAR(120)` | `NOT NULL` | — | Human-readable event name |
| `location` | `VARCHAR(200)` | | — | Venue name and city |
| `start_date` | `DATE` | `NOT NULL` | — | First day of competition |
| `end_date` | `DATE` | `NOT NULL` | — | Last day of competition |
| `season_year` | `SMALLINT` | `NOT NULL` | — | FRC season year (e.g., `2025`) |
| `perspective_matrix` | `JSONB` | | `NULL` | Serialised 3×3 homography matrix for venue camera calibration |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL` | `now()` | Row creation time |

**Indexes:**
- `UNIQUE (tba_event_key)` — TBA event key lookup
- `INDEX (season_year)` — filter events by season

**Example Data:**
```json
{
  "event_id": 1,
  "tba_event_key": "2025calb",
  "name": "2025 Los Angeles Regional",
  "location": "Walter Pyramid, Long Beach, CA",
  "start_date": "2025-03-13",
  "end_date": "2025-03-16",
  "season_year": 2025,
  "perspective_matrix": [[1.2,0,0],[0,1.1,0],[0,0,1]],
  "created_at": "2025-01-15T00:00:00Z"
}
```

### 2.5 ROBOT_PERFORMANCE

The central fact table linking a specific team to a specific match slot. All per-robot statistics and CV tracking data reference this table.

**Table:** `robot_performance`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `perf_id` | `SERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `match_id` | `INTEGER` | `FK → MATCH(match_id) NOT NULL` | — | Parent match |
| `team_id` | `INTEGER` | `FK → TEAM(team_id) NOT NULL` | — | Team that played |
| `alliance_id` | `INTEGER` | `FK → ALLIANCE(alliance_id) NOT NULL` | — | Which alliance |
| `alliance_position` | `SMALLINT` | `NOT NULL CHECK (alliance_position IN (1,2,3))` | — | Robot position (1, 2, or 3) in the alliance |
| `track_id` | `SMALLINT` | | `NULL` | DeepSORT track ID assigned during CV processing |
| `auto_score` | `SMALLINT` | `NOT NULL` | `0` | Points scored in Autonomous phase |
| `teleop_score` | `SMALLINT` | `NOT NULL` | `0` | Points scored in Tele-op phase |
| `endgame_score` | `SMALLINT` | `NOT NULL` | `0` | Points scored in Endgame phase |
| `total_score_contribution` | `SMALLINT` | `GENERATED ALWAYS AS (auto_score + teleop_score + endgame_score) STORED` | computed | Computed total contribution |
| `fouls_drawn` | `SMALLINT` | `NOT NULL` | `0` | Fouls drawn against opponents |
| `fouls_committed` | `SMALLINT` | `NOT NULL` | `0` | Fouls committed by this robot |
| `no_show` | `BOOLEAN` | `NOT NULL` | `false` | Robot did not appear on field |
| `disabled` | `BOOLEAN` | `NOT NULL` | `false` | Robot was disabled mid-match |

**Constraints:**
- `UNIQUE (match_id, team_id)` — a team plays at most once per match

**Indexes:**
- `INDEX (team_id, match_id)` — team match history queries
- `INDEX (alliance_id)` — alliance roster lookup

**Example Data:**
```json
{
  "perf_id": 301,
  "match_id": 101,
  "team_id": 1,
  "alliance_id": 201,
  "alliance_position": 1,
  "track_id": 3,
  "auto_score": 18,
  "teleop_score": 42,
  "endgame_score": 15,
  "total_score_contribution": 75,
  "fouls_drawn": 1,
  "fouls_committed": 0,
  "no_show": false,
  "disabled": false
}
```

## 3. Authentication Models (Phase 1, Tier 3)

### 3.1 USER

Stores authenticated user accounts with role-based access control.

**Table:** `user`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `user_id` | `SERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `username` | `VARCHAR(80)` | `UNIQUE NOT NULL` | — | Login username |
| `email` | `VARCHAR(200)` | `UNIQUE NOT NULL` | — | Contact email address |
| `hashed_password` | `VARCHAR(200)` | `NOT NULL` | — | bcrypt hash of the user's password |
| `team_id` | `INTEGER` | `FK → TEAM(team_id)` | `NULL` | Team affiliation (optional) |
| `role` | `VARCHAR(20)` | `NOT NULL CHECK (role IN ('SCOUT','COACH','TEAM_ADMIN','SYSTEM_ADMIN'))` | `'SCOUT'` | Role determining access level |
| `is_active` | `BOOLEAN` | `NOT NULL` | `true` | Whether the account is enabled |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL` | `now()` | Account creation time |
| `last_login` | `TIMESTAMPTZ` | | `NULL` | Most recent successful login |

**Role Definitions:**

| Role | Permissions |
|---|---|
| `SCOUT` | Read events/teams/matches; submit scouting observations |
| `COACH` | All SCOUT permissions; read all team analytics; manage team's scouting data |
| `TEAM_ADMIN` | All COACH permissions; manage team's user accounts |
| `SYSTEM_ADMIN` | Full access; manage all users; trigger TBA syncs; manage events |

**Password Security:**
- Passwords are hashed using **bcrypt** (via `passlib`) before storage
- Plaintext passwords are never persisted or logged
- JWT access tokens are issued on successful login using **python-jose** (HS256, configurable expiry via `JWT_EXPIRE_MINUTES` env var)

**Indexes:**
- `UNIQUE (username)` — login lookup
- `UNIQUE (email)` — duplicate account prevention
- `INDEX (team_id)` — team member queries

**Example Data:**
```json
{
  "user_id": 1,
  "username": "coach_jones",
  "email": "jones@team1678.org",
  "hashed_password": "$2b$12$...",
  "team_id": 1,
  "role": "COACH",
  "is_active": true,
  "created_at": "2025-01-20T08:00:00Z",
  "last_login": "2025-03-14T09:45:00Z"
}
```

## 4. Data Collection Models (Phase 1, Tier 5)

### 4.1 SCOUTING_OBSERVATION

Stores manual observations submitted by human scouts during or after a match.

**Table:** `scouting_observation`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `observation_id` | `SERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `team_id` | `INTEGER` | `FK → TEAM(team_id) NOT NULL` | — | Team being scouted |
| `match_id` | `INTEGER` | `FK → MATCH(match_id) NOT NULL` | — | Match being observed |
| `scout_id` | `INTEGER` | `FK → USER(user_id) NOT NULL` | — | Scout who submitted the observation |
| `notes` | `TEXT` | | `NULL` | Free-text qualitative notes |
| `rating` | `SMALLINT` | `CHECK (rating BETWEEN 1 AND 5)` | `NULL` | Overall performance rating (1–5) |
| `actions` | `JSONB` | | `NULL` | Structured action log (see schema below) |
| `submitted_at` | `TIMESTAMPTZ` | `NOT NULL` | `now()` | Submission timestamp |

**`actions` JSONB Schema:**
```json
{
  "auto": {
    "mobility": true,
    "notes_scored": 3,
    "speaker_scored": 1
  },
  "teleop": {
    "notes_scored": 8,
    "speaker_scored": 2,
    "amp_scored": 6
  },
  "endgame": {
    "climbed": true,
    "harmonized": false,
    "trap_scored": false
  },
  "defense_played": false,
  "breakdown": false
}
```

**Indexes:**
- `INDEX (team_id, match_id)` — team-match observation lookup
- `INDEX (scout_id)` — scout's submission history

**Example Data:**
```json
{
  "observation_id": 501,
  "team_id": 1,
  "match_id": 101,
  "scout_id": 1,
  "notes": "Very fast cycles, struggled with endgame climb",
  "rating": 4,
  "actions": {"auto": {"mobility": true, "notes_scored": 3}, "teleop": {"notes_scored": 8}},
  "submitted_at": "2025-03-14T11:05:00Z"
}
```

## 5. Sync & Audit Models (Phase 1, Tier 4)

### 5.1 SYNC_LOG

Provides an immutable audit trail of every data synchronisation from The Blue Alliance API or other external sources.

**Table:** `sync_log`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `sync_id` | `BIGSERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `sync_type` | `VARCHAR(30)` | `NOT NULL` | — | Category of sync: `event`, `team`, `match`, `robot_performance` |
| `resource_id` | `VARCHAR(50)` | `NOT NULL` | — | The TBA key or internal ID of the synced resource |
| `status` | `VARCHAR(20)` | `NOT NULL` | `'success'` | Outcome: `success`, `partial`, `failed` |
| `records_created` | `INTEGER` | `NOT NULL` | `0` | Number of rows inserted |
| `records_updated` | `INTEGER` | `NOT NULL` | `0` | Number of rows updated |
| `old_values` | `JSONB` | | `NULL` | Snapshot of values before the update |
| `new_values` | `JSONB` | | `NULL` | Snapshot of values after the update |
| `error_message` | `TEXT` | | `NULL` | Error description when status is `failed` |
| `sync_timestamp` | `TIMESTAMPTZ` | `NOT NULL` | `now()` | When the sync ran |
| `triggered_by` | `INTEGER` | `FK → USER(user_id)` | `NULL` | User who triggered the sync (NULL = scheduled/daemon) |

**Purpose:** The SYNC_LOG table enables:
- Debugging data discrepancies between ScouterFRC and TBA
- Auditing which admin triggered a manual sync
- Detecting partial sync failures to re-run only affected records
- Providing a diff history when TBA corrects match results post-event

**Indexes:**
- `INDEX (sync_type, resource_id)` — lookup history for a specific resource
- `INDEX (sync_timestamp DESC)` — recent sync history dashboard

## 6. Phase 2 Models

> **Note:** Phase 2 models are populated by the Computer Vision pipeline (YOLOv8 + DeepSORT). They should not be created in the database until Phase 2 development begins.

### 6.1 SCORING_EVENT

Individual scoring moments detected by the CV event classifier during video analysis.

**Table:** `scoring_event`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `scoring_event_id` | `SERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `perf_id` | `INTEGER` | `FK → ROBOT_PERFORMANCE(perf_id) NOT NULL` | — | Robot this event belongs to |
| `event_type` | `VARCHAR(30)` | `NOT NULL` | — | e.g., `score_attempt`, `score_success`, `pickup`, `climb_start`, `climb_success`, `penalty` |
| `phase` | `VARCHAR(15)` | `NOT NULL CHECK (phase IN ('auto','teleop','endgame'))` | — | Game phase when event occurred |
| `frame_number` | `INTEGER` | `NOT NULL` | — | Frame index in source video |
| `timestamp_s` | `NUMERIC(7,3)` | `NOT NULL` | — | Seconds from match start |
| `field_x_m` | `NUMERIC(6,3)` | | `NULL` | Robot x-position on field (metres) |
| `field_y_m` | `NUMERIC(6,3)` | | `NULL` | Robot y-position on field (metres) |
| `confidence` | `NUMERIC(4,3)` | `CHECK (confidence BETWEEN 0 AND 1)` | `NULL` | CV classifier confidence score |
| `notes` | `TEXT` | | `NULL` | Optional manual annotation override |

**Indexes:**
- `INDEX (perf_id, phase, event_type)` — phase-breakdown queries
- `INDEX (perf_id, timestamp_s)` — time-ordered event replay

### 6.2 MOVEMENT_TRACK

Raw positional samples for a robot throughout a match, emitted at each processed video frame by the CV pipeline.

**Table:** `movement_track`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `track_id` | `BIGSERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `perf_id` | `INTEGER` | `FK → ROBOT_PERFORMANCE(perf_id) NOT NULL` | — | Robot this sample belongs to |
| `frame_number` | `INTEGER` | `NOT NULL` | — | Source video frame index |
| `timestamp_s` | `NUMERIC(7,3)` | `NOT NULL` | — | Seconds from match start |
| `field_x_m` | `NUMERIC(6,3)` | `NOT NULL` | — | Robot x-position on field (metres) |
| `field_y_m` | `NUMERIC(6,3)` | `NOT NULL` | — | Robot y-position on field (metres) |
| `pixel_x` | `SMALLINT` | | `NULL` | Original pixel x (pre-transform) |
| `pixel_y` | `SMALLINT` | | `NULL` | Original pixel y (pre-transform) |
| `track_confidence` | `NUMERIC(4,3)` | `CHECK (track_confidence BETWEEN 0 AND 1)` | `NULL` | DeepSORT track confidence |
| `phase` | `VARCHAR(15)` | `NOT NULL CHECK (phase IN ('auto','teleop','endgame'))` | — | Game phase |

**Indexes:**
- `INDEX (perf_id, timestamp_s)` — primary time-ordered traversal for cycle detection
- `INDEX (perf_id, frame_number)` — frame-based video seek

**Performance Note:** For high-volume deployments (≥ 100 matches/event × 6 robots × ~1,500 frames/robot = ~900,000 rows/event), partition this table by `match_id` or convert to a TimescaleDB hypertable keyed on `timestamp_s`.

### 6.3 PHASE_STAT

Aggregated per-phase statistics for a robot derived from `SCORING_EVENT` and `MOVEMENT_TRACK` data.

**Table:** `phase_stat`

| Column | Type | Constraints | Default | Description |
|---|---|---|---|---|
| `stat_id` | `SERIAL` | `PRIMARY KEY` | auto | Internal surrogate key |
| `perf_id` | `INTEGER` | `FK → ROBOT_PERFORMANCE(perf_id) NOT NULL` | — | Parent performance record |
| `phase` | `VARCHAR(15)` | `NOT NULL CHECK (phase IN ('auto','teleop','endgame'))` | — | Game phase |
| `attempts` | `SMALLINT` | `NOT NULL` | `0` | Scoring attempts in this phase |
| `successes` | `SMALLINT` | `NOT NULL` | `0` | Successful scores in this phase |
| `success_rate` | `NUMERIC(5,4)` | `GENERATED ALWAYS AS (CASE WHEN attempts = 0 THEN NULL ELSE successes::numeric / attempts END) STORED` | computed | Computed success percentage |
| `avg_cycle_time_s` | `NUMERIC(6,2)` | | `NULL` | Mean cycle time (seconds) |
| `min_cycle_time_s` | `NUMERIC(6,2)` | | `NULL` | Fastest cycle observed |
| `max_cycle_time_s` | `NUMERIC(6,2)` | | `NULL` | Slowest cycle observed |
| `distance_travelled_m` | `NUMERIC(8,2)` | | `NULL` | Total distance from movement track |

**Constraints:**
- `UNIQUE (perf_id, phase)` — exactly one row per phase per performance record

| `TEAM` | `TEAM_ORGANIZATION` | M:M | `team_organization.team_id` | `ON DELETE CASCADE` |
| `ORGANIZATION` | `TEAM_ORGANIZATION` | M:M | `team_organization.org_id` | `ON DELETE CASCADE` |

### 9.2 Cascade Behavior Rationale

- **`ON DELETE RESTRICT` on TEAM and EVENT:** Prevents accidental deletion of foundational data that has associated match history.
- **`ON DELETE CASCADE` on ALLIANCE and ROBOT_PERFORMANCE:** Child CV data (MOVEMENT_TRACK, SCORING_EVENT) should be cleaned up automatically when a performance record is removed.
- **`ON DELETE SET NULL` on USER references:** Preserves SYNC_LOG and USER rows even if a user account is deleted (audit trail must be retained).

## 11. Indexes

### 11.1 Index Strategy

ScouterFRC's index strategy follows three principles:
1. **Support every FK with an index** to avoid sequential scans on join operations.
2. **Composite indexes** align with the most common query patterns (event schedule, team history, phase analytics).
3. **Defer MOVEMENT_TRACK partitioning** until row volume exceeds 5 million rows; until then, the `(perf_id, timestamp_s)` index is sufficient.

### 11.2 Indexes by Table

**EVENT**
```sql
CREATE UNIQUE INDEX uq_event_tba_key ON event (tba_event_key);
CREATE INDEX idx_event_season ON event (season_year);
```

**TEAM**
```sql
CREATE UNIQUE INDEX uq_team_number ON team (team_number);
```

**MATCH**
```sql
CREATE UNIQUE INDEX uq_match_tba_key ON match (tba_match_key);
CREATE INDEX idx_match_event_schedule ON match (event_id, match_type, match_number);
CREATE INDEX idx_match_processing_status ON match (processing_status) WHERE processing_status != 'complete';
```

**ALLIANCE**
```sql
CREATE UNIQUE INDEX uq_alliance_match_color ON alliance (match_id, color);
```

**ROBOT_PERFORMANCE**
```sql
CREATE UNIQUE INDEX uq_perf_match_team ON robot_performance (match_id, team_id);
CREATE INDEX idx_perf_team_match ON robot_performance (team_id, match_id);
CREATE INDEX idx_perf_alliance ON robot_performance (alliance_id);
```

**USER**
```sql
CREATE UNIQUE INDEX uq_user_username ON "user" (username);
CREATE UNIQUE INDEX uq_user_email ON "user" (email);

```

**SCOUTING_OBSERVATION**


**SYNC_LOG**

**SCORING_EVENT** *(Phase 2)*


**MOVEMENT_TRACK** *(Phase 2)*


**PHASE_STAT** *(Phase 2)*


### 11.3 Query Optimization Notes

- **Heatmap queries** (`SELECT field_x_m, field_y_m FROM movement_track WHERE perf_id = ?`) benefit most from the `(perf_id, timestamp_s)` index since the query engine can do an index-only scan.
- **Event dashboard** (`SELECT t.team_number, SUM(rp.total_score_contribution) ... GROUP BY team`) joins EVENT → MATCH → ROBOT_PERFORMANCE → TEAM using all FK indexes.
- **Alliance builder** (`SELECT ... FROM robot_performance WHERE alliance_id IN (?)`) uses `idx_perf_alliance` for fast alliance roster retrieval.

## 12. Data Types & Sizing

### 12.1 Column Type Choices and Rationale

| Type | Used For | Rationale |
|---|---|---|
| `SERIAL` / `BIGSERIAL` | Primary keys | Auto-incrementing; `BIGSERIAL` for high-volume MOVEMENT_TRACK |
| `SMALLINT` | FRC-domain numerics (team number, scores, frame coords) | FRC teams ≤ 9999; scores ≤ 32767 — fits in 2 bytes |
| `INTEGER` | Foreign keys, frame numbers | 4-byte int for FK references |
| `VARCHAR(n)` | Keys and names with bounded length | Avoids unlimited TEXT for indexed/constrained columns |
| `TEXT` | Free text (notes, error messages) | Unbounded text; not indexed |
| `DATE` | Event dates | 4-byte storage; no timezone needed for calendar dates |
| `TIMESTAMPTZ` | Audit timestamps | UTC-normalised; avoids DST bugs |
| `NUMERIC(p,s)` | Scores, confidence, coordinates | Exact decimal arithmetic; avoids float rounding |
| `BOOLEAN` | Flags (no_show, disabled, won) | 1 byte; clear semantics |
| `JSONB` | Perspective matrix, action payloads, sync diffs | Binary JSON for fast key lookup; supports GIN indexing |

## 13. Migration Strategy

### 13.1 Alembic Migration Naming Convention

```
<sequence>_<phase>_<tier>_<short_description>.py
```

**Examples:**
```
0001_p1t1_initial_schema.py
0002_p1t3_add_user_table.py
0003_p1t4_add_sync_log.py
0004_p1t5_add_scouting_observation.py
0005_p2t2_add_movement_track.py
0006_p2t3_add_scoring_event_phase_stat.py
0007_p3t1_add_organization.py
```

### 13.2 Phase-Based Migrations

**Phase 1 Migrations (Tier 1–5):**
- `0001` — Core tables: EVENT, TEAM, MATCH, ALLIANCE, ROBOT_PERFORMANCE
- `0002` — Authentication: USER
- `0003` — Audit: SYNC_LOG
- `0004` — Data collection: SCOUTING_OBSERVATION

**Phase 2 Migrations (Tier 2–3):**
- `0005` — CV output: MOVEMENT_TRACK, SCORING_EVENT
- `0006` — Analytics: PHASE_STAT; add `track_id` to ROBOT_PERFORMANCE

**Phase 3 Migrations (Tier 1):**
- `0007` — Multi-org: ORGANIZATION, TEAM_ORGANIZATION

### 13.3 Running Migrations

```bash
# Upgrade to latest
alembic upgrade head

# Upgrade one step
alembic upgrade +1

# Downgrade one step
alembic downgrade -1

# Downgrade to clean slate
alembic downgrade base

# Show current revision
alembic current

# Show migration history
alembic history --verbose
```

### 13.4 Rollback Procedures

Each migration file must implement a complete `downgrade()` function:
- `CREATE TABLE` is paired with `DROP TABLE IF EXISTS`
- `ADD COLUMN` is paired with `DROP COLUMN`
- `CREATE INDEX` is paired with `DROP INDEX`
- `CREATE UNIQUE CONSTRAINT` is paired with `DROP CONSTRAINT`

**Zero-downtime column additions:** New nullable columns can be added without a lock. Non-nullable columns require a multi-step migration: (1) add nullable, (2) backfill, (3) add NOT NULL constraint.

### 13.5 Data Preservation During Migrations

- Never drop a column in the same migration that removes its data. Use a two-step approach:
  1. Migration N: mark column deprecated (rename to `_deprecated_<col>`)
  2. Migration N+1 (next release): drop the deprecated column after confirming no usage
- Use `alembic stamp <revision>` to mark existing databases without running migrations when first introducing Alembic to a pre-existing schema.

### 14.2 TEAM Queries

**Get all teams at an event:**
```sql
SELECT DISTINCT t.team_id, t.team_number, t.team_name, t.city, t.state_prov
FROM team t
JOIN robot_performance rp ON rp.team_id = t.team_id
JOIN match m ON m.match_id = rp.match_id
WHERE m.event_id = 1
ORDER BY t.team_number;
```

```sql
SELECT m.tba_match_key, m.match_type, m.match_number,
       a.color, rp.auto_score, rp.teleop_score,
       rp.endgame_score, rp.total_score_contribution
FROM robot_performance rp
JOIN match m ON m.match_id = rp.match_id
JOIN alliance a ON a.alliance_id = rp.alliance_id
WHERE rp.team_id = 1
ORDER BY m.match_type, m.match_number;
```

### 14.3 MATCH Queries

**Get full match details with alliances:**
```sql
SELECT m.tba_match_key, m.match_type, m.match_number, m.played_at,
       a.color, a.total_score, a.won,
       t.team_number, rp.alliance_position,
       rp.auto_score, rp.teleop_score, rp.endgame_score
FROM match m
JOIN alliance a ON a.match_id = m.match_id
JOIN robot_performance rp ON rp.alliance_id = a.alliance_id
JOIN team t ON t.team_id = rp.team_id
WHERE m.match_id = 101
ORDER BY a.color, rp.alliance_position;
```

```sql
SELECT match_id, tba_match_key, video_url
FROM match
WHERE processing_status = 'pending'
  AND video_url IS NOT NULL
ORDER BY played_at ASC
LIMIT 10;
```

### 14.4 ROBOT_PERFORMANCE Aggregation

**Team scoring averages across an event:**
```sql
SELECT t.team_number, t.team_name,
       ROUND(AVG(rp.auto_score), 2)    AS avg_auto,
       ROUND(AVG(rp.teleop_score), 2)  AS avg_teleop,
       ROUND(AVG(rp.endgame_score), 2) AS avg_endgame,
       ROUND(AVG(rp.total_score_contribution), 2) AS avg_total,
       COUNT(*) AS matches_played
FROM robot_performance rp
JOIN team t ON t.team_id = rp.team_id
JOIN match m ON m.match_id = rp.match_id
WHERE m.event_id = 1
  AND rp.no_show = false
GROUP BY t.team_id, t.team_number, t.team_name
ORDER BY avg_total DESC;
```

**Alliance projected score:**
```sql
SELECT a.color,
       SUM(rp.total_score_contribution) AS alliance_total
FROM alliance a
JOIN robot_performance rp ON rp.alliance_id = a.alliance_id
WHERE a.match_id = 101
GROUP BY a.alliance_id, a.color;
```

### 14.5 Phase 2 Analytics Queries

**Team heatmap data (MOVEMENT_TRACK):**
```sql
SELECT mt.field_x_m, mt.field_y_m, COUNT(*) AS dwell_frames
FROM movement_track mt
JOIN robot_performance rp ON rp.perf_id = mt.perf_id
JOIN match m ON m.match_id = rp.match_id
WHERE rp.team_id = 1
  AND m.event_id = 1
GROUP BY mt.field_x_m, mt.field_y_m
ORDER BY dwell_frames DESC;
```

**Phase success rates (PHASE_STAT):**
```sql
SELECT t.team_number,
       ps.phase,
       ROUND(AVG(ps.attempts), 1)    AS avg_attempts,
       ROUND(AVG(ps.successes), 1)   AS avg_successes,
       ROUND(AVG(ps.success_rate), 4) AS avg_success_rate,
       ROUND(AVG(ps.avg_cycle_time_s), 2) AS avg_cycle_s
FROM phase_stat ps
JOIN robot_performance rp ON rp.perf_id = ps.perf_id
JOIN team t ON t.team_id = rp.team_id
JOIN match m ON m.match_id = rp.match_id
WHERE m.event_id = 1
GROUP BY t.team_id, t.team_number, ps.phase
ORDER BY t.team_number, ps.phase;
```

### 14.6 SCOUTING_OBSERVATION Queries

**Scout observations for a team at an event:**

## 15. Backup & Recovery

### 15.1 Backup Strategy

| Tier | Frequency | Method | Retention |
|---|---|---|---|
| **Full backup** | Daily (off-peak) | `pg_dump --format=custom` | 30 days |
| **Incremental WAL** | Continuous | WAL archiving to S3 (`archive_command`) | 7 days |
| **Pre-migration snapshot** | Before each `alembic upgrade` | `pg_dump` to S3 with migration tag | Indefinitely |
| **Event export** | After each event completes | JSON export via API endpoint | Indefinitely |

**Backup Command:**
```bash
pg_dump \
  --format=custom \
  --compress=9 \
  --file="scoutfrc_$(date +%Y%m%d_%H%M%S).dump" \
  $DATABASE_URL
```

### 15.2 Data Retention Policies

| Data Category | Retention | Rationale |
|---|---|---|
| Event/Team/Match metadata | Indefinitely | Historical records are valuable for multi-season analysis |
| MOVEMENT_TRACK rows | 2 seasons | High-volume; older data can be archived to S3 Glacier |
| SCORING_EVENT rows | 3 seasons | Needed for cross-season ML model training |
| SCOUTING_OBSERVATION | Indefinitely | Human-entered data is irreplaceable |
| SYNC_LOG entries | 1 year | Audit trail for compliance; older entries can be purged |
| USER accounts | Until manual deletion | GDPR/COPPA: users may request account removal |

### 15.3 Recovery Procedures

**Full restore from dump:**
```bash
pg_restore \
  --clean \
  --if-exists \
  --dbname=$DATABASE_URL \
  scoutfrc_20250314_000000.dump
```

**Point-in-time recovery (with WAL archiving):**
1. Restore latest base backup
2. Apply WAL segments up to the target recovery time
3. Set `recovery_target_time` in `postgresql.conf`
4. Start PostgreSQL in recovery mode

**Single-table restore:**
```bash
pg_restore \
  --table=scouting_observation \
  --dbname=$DATABASE_URL \
  scoutfrc_20250314_000000.dump
```

## 16. Security Considerations

### 16.1 Data Encryption

| Layer | Mechanism |
|---|---|
| **In transit** | TLS 1.3 for all connections (enforce `sslmode=require` in connection strings) |
| **At rest** | AWS EBS volume encryption (AES-256) for PostgreSQL data directory |
| **Backup files** | AES-256 encryption of S3 bucket objects via AWS KMS |
| **JWT tokens** | HS256 signing with a 256-bit secret stored in environment variables |

### 16.2 Access Control

```
┌─────────────────┬──────────────────────────────────────────────────────────┐
│ Role            │ Database Permissions                                      │
├─────────────────┼──────────────────────────────────────────────────────────┤
│ app_read        │ SELECT on all tables                                      │
│ app_write       │ INSERT, UPDATE on observation, sync_log                  │
│ app_admin       │ INSERT, UPDATE, DELETE on all tables                     │
│ migration_role  │ DDL permissions (CREATE, DROP, ALTER) — used by Alembic  │
│ backup_role     │ SELECT on all tables, CONNECT — used by pg_dump          │
└─────────────────┴──────────────────────────────────────────────────────────┘
```

### 16.3 Sensitive Data Handling

| Data | Handling |
|---|---|
| **Passwords** | bcrypt hashed (cost factor ≥ 12) before storage; never logged |
| **JWT secrets** | Stored in environment variables / AWS Secrets Manager; not in code |
| **TBA API key** | Stored in environment variables; never committed to version control |
| **User emails** | Required for account recovery; not displayed publicly |
| **S3 credentials** | IAM role-based access; no long-lived access keys in code |

### 16.4 PII Data Considerations

- **Email addresses** are PII under GDPR/CCPA. The `USER.email` column should be masked in non-production environments.
- **Usernames** may be considered PII depending on jurisdiction. Anonymize in data exports.
- **SCOUTING_OBSERVATION.notes** may contain PII if scouts include real names. Implement a data export and deletion mechanism to support user data removal requests.
- FRC team members are often minors — do not expose `USER` data via public API endpoints and ensure age-appropriate consent flows for account creation.

### 16.5 SQL Injection Prevention

- All queries use SQLAlchemy ORM bound parameters; raw SQL is never constructed via string concatenation.
- Input validation is enforced at the Pydantic schema layer before reaching the database layer.

## 17. Scalability & Performance

### 17.1 Current Capacity Estimates

| Metric | Capacity |
|---|---|
| Events per season | 5–50 |
| Concurrent users during an event | 10–50 |
| Matches processed per hour | ~12 (limited by CV GPU) |
| Peak API requests/second | ~50 (dashboard polling) |

### 17.2 Growth Handling

| Growth Trigger | Strategy |
|---|---|
| > 5M MOVEMENT_TRACK rows | Convert to TimescaleDB hypertable partitioned by `match_id` |
| > 10K concurrent users | Add read replicas; move analytics queries to replica |
| > 100 events/season | Enable connection pooling via PgBouncer |
| Dashboard latency > 500 ms | Pre-compute event leaderboards into Redis on each TBA sync |

### 17.3 Partitioning Strategy for MOVEMENT_TRACK

```sql
-- Range partition by event year (when TimescaleDB is not available)
CREATE TABLE movement_track (
    ...
) PARTITION BY RANGE (perf_id);  -- use match year via FK join

-- Or with TimescaleDB:
SELECT create_hypertable('movement_track', 'timestamp_s',
       chunk_time_interval => 150);  -- 150-second chunks = one match
```

### 17.4 Redis Caching

| Cache Key | TTL | Content |
|---|---|---|
| `event:{event_id}:leaderboard` | 5 min | Pre-sorted team rankings |
| `team:{team_id}:heatmap:{event_id}` | 10 min | Serialised 2D density array |
| `match:{match_id}:alliance_score` | 1 match cycle | Alliance projected scores |
| `user:{user_id}:session` | JWT expiry | User session context |

### 17.5 Archive Strategy

After a season closes:
1. Export full season data via `pg_dump --table` for each major table.
2. Upload compressed archives to S3 Glacier (low-cost, infrequent access).
3. Delete `MOVEMENT_TRACK` rows older than 2 seasons from the live database.
4. Keep all other tables (EVENT, TEAM, MATCH, ROBOT_PERFORMANCE) indefinitely for historical comparison.

## 18. Future Considerations

### 18.1 Phase 3 Multi-Tenancy Implications

The `ORGANIZATION` and `TEAM_ORGANIZATION` tables introduced in Phase 3 create the foundation for data isolation:
- Row-level security (RLS) policies in PostgreSQL can restrict data access by `org_id`
- A middleware layer must inject the `org_id` from the JWT claim into each database session
- Cross-organization analytics require a separate elevated privilege role

### 18.2 Data Isolation Requirements

| Isolation Level | Mechanism |
|---|---|
| Team-level (current) | Application-layer filtering by `team_id` |
| Organization-level (Phase 3) | PostgreSQL RLS policies on `org_id` |
| System-level | Separate PostgreSQL schemas per large organization (if needed) |

### 18.3 API Caching Strategy

As the system scales beyond Phase 1:
- **Redis** caches hot read paths (leaderboards, heatmaps, alliance builder projections)
- **CDN caching** for static analytics exports (end-of-event PDF reports)
- **ETags** on GET endpoints to support conditional requests and reduce bandwidth

### 18.4 Timeline for Schema Evolution

| Milestone | Schema Change |
|---|---|
| Phase 1 complete | 8 tables; ~300 KB per event |
| Phase 2 complete | +3 tables (MOVEMENT_TRACK, SCORING_EVENT, PHASE_STAT); ~74 MB per event |
| Phase 3 complete | +2 tables (ORGANIZATION, TEAM_ORGANIZATION); multi-tenant RLS |
| Season 2 | Consider TimescaleDB hypertable for MOVEMENT_TRACK |
| Season 3+ | Evaluate read replicas and connection pooling |

### 18.5 Potential Schema Extensions (Not Yet Planned)

- **GAME_PIECE_EVENT** — track game piece pickups and scoring locations per game year
- **STRATEGY_NOTE** — structured pre-match strategy notes from coaches
- **ML_PREDICTION** — store model predictions and confidence scores for alliance selection
